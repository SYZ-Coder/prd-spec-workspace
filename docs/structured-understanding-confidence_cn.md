# 结构化理解与可信度说明

这份文档说明：本项目如何通过“结构化理解”和“可信度透明”这两条设计原则，提升多模态需求识别与规格转换的准确度。

先明确边界：

- 本项目不是通用知识图谱平台
- 本项目不是代码图谱或代码理解工具
- 本项目的目标始终是“多模态需求识别 -> 结构化 DSL -> 规格与执行产物”

## 1. 结构化理解

面对 PRD、截图、备注、流程图、上下文说明等多源输入，平台不会只把它们当作一堆文本顺序处理，而是先统一收口到一个可查询、可校验、可追溯的结构层。

在本项目里，这一层主要体现为：

- `working/raw-dsl.json`
- `working/merged-dsl.json`
- `working/evidence-map.md`
- `merged-dsl.json` 中的 `knowledge_graph`

这里的 `knowledge_graph` 不是为了做图谱产品，而是为了把需求中的核心对象和关系显式化，例如：

- 页面
- 动作
- 状态
- 角色
- 接口
- 规则
- 依赖
- 页面间流转

## 2. 可信度透明

需求识别天然带有不确定性，尤其是在只有截图、备注不完整、流程证据冲突时。

所以平台不应该只给一个“看起来完整”的结果，还要告诉用户：

- 哪些是高可信度识别
- 哪些是根据弱证据推断的
- 哪些仍然需要人工确认
- 哪些冲突已经进入 `unknowns`

在本项目里，这一层主要体现为：

- `confidence_summary`
- 页面 / 动作 / 流转级别的 `confidence`
- 对应条目的 `evidence`
- [evidence-map.md](working/evidence-map.md)
- [validation-report.md](working/validation-report.md)

## 3. 在本平台中的落地方式

- 所有多模态识别结果先进入 DSL 中间层
- 证据和结论尽量绑定输出
- validation 负责兜住结构风险和部分语义风险
- unknowns 明确保留，不把待确认项伪装成事实

## 4. 这不是在做什么

为了避免误解，这里明确说明本项目没有在做的事：

- 不是做一个通用知识图谱产品
- 不是做代码仓库理解平台
- 不是做网页抓取和知识问答平台
- 不是把所有需求都自动推断成完整终稿

## 5. 对准确度和可信度的实际帮助

这套思路落地后，本平台在两个方向上更稳：

- 准确度：页面、动作、状态、接口、角色不再只是散落在文稿里，而是进入统一结构层接受归并和校验
- 可信度：用户可以看到证据来源、低可信度项清单，以及“已确认事实 / 结构化推断 / 待确认项”的分层

## 6. 后续可以继续增强的方向

- 扩大 `knowledge_graph` 的关系表达能力
- 让 `evidence-map.md` 更便于人工复核
- 继续增强 validation 的语义风险识别
- 让截图、流程图、上下文说明之间的证据对齐更稳定

## 7. 总结

本项目坚持两条关键原则：

- 把需求理解过程结构化
- 把识别结果的可信度透明化

平台的最终定位仍然是：

- 面向多模态需求输入
- 面向规格生成与执行衔接
- 面向产品、测试、研发和 AI 协作的需求结构化平台
